The Gap

AI agents are being pushed into production faster than security and compliance can keep up. Most enterprises still rely on shared API keys, over‑privileged access, and unsigned app logs when something goes wrong. There is no clear way to answer:

  • Which agent acted?
  • Under which policy?
  • Can we prove this to an auditor or insurer without trusting our own logs?

At the same time, certifications like AIUC‑1 and ISO 42001, obligations under the EU AI Act, and emerging AI insurance products all demand independent, tamper‑evident evidence of how agents behave in production.

The Bet

AgentMint started life as an AIUC‑1 evidence layer: a cryptographic notary that turns AI agent actions into signed receipts you can hand to auditors. The prototype works in production against ElevenLabs and Claude, and produces receipts that verify cleanly with OpenSSL and a public timestamp authority.

Conversations with CISOs, AI governance leads, and security VCs converged on a deeper, more urgent entry point:

Identity and access are the first problem, and evidence is the exhaust. Enterprises need to give each agent a first‑class identity, enforce least‑privilege on every action, and only then worry about turning those decisions into audit‑ready evidence.

The pivot is to make AgentMint the agent identity gateway: where agents get identities, policies are enforced, and cryptographic receipts are generated as a byproduct.

What AgentMint Does

1. Issue Agent Identities

AgentMint creates cryptographic identities for each AI agent or workflow (non‑human identities), tied to owners, purpose, and risk tier. These identities are used to authenticate agents to downstream tools and APIs.

2. Enforce Least‑Privilege

Policies define which tools, APIs, data scopes, and environments each agent can touch. The gateway allows or blocks actions based on these policies, and supports checkpoint requirements for sensitive operations.

3. Emit Cryptographic Receipts

Every allowed or denied action generates a tamper‑evident receipt:

  • Agent identity and policy version
  • Resource / tool, environment, decision (allow / deny)
  • Content and source log hashes (no raw data)
  • Ed25519 signature and RFC 3161 timestamp
  • Explicit mapping to AIUC‑1 / ISO 42001 controls

Receipts are designed to be verifiable with openssl and a public TSA certificate—no AgentMint infrastructure is required for verification.

4. Package Evidence for Audits and Incidents

Receipts accumulate in an append‑only store and can be exported as “evidence packs” for:

  • AIUC‑1 re‑certification
  • ISO 42001 and NIST AI RMF assessments
  • EU AI Act documentation requests
  • Insurance underwriting and incident forensics

Architecture

AgentMint is implemented as two cooperating layers in a Python SDK:

  • Gatekeeper (core.py) – authorization before the action. Issues agent identities, evaluates policies, and decides whether an action should be allowed. Produces structured decision events.
  • Notary (notary.py) – evidence after the action. Takes decision events and runtime context, then builds, signs, and timestamps receipts.
agentmint/
├── core.py            # Gatekeeper: scoped delegation, checkpoints, replay protection
├── notary.py          # Notary: passive receipt generation, policy evaluation
├── anchor.py          # RFC 3161 timestamping (FreeTSA + DigiCert fallback)
├── commitment.py      # SHA-256 commitment scheme (hash-only receipts)
├── batch.py           # Batch mode: scenario loading, execution, aggregation
├── export.py          # Evidence ZIP packaging
├── keystore.py        # Ed25519 key persistence (generate once, load thereafter)
├── receipt_store.py   # JSONL append-only receipt persistence
└── ...

The current prototype focuses on the Notary: running alongside real agents, observing responses, applying policy, and generating receipts. The Gatekeeper is being extended into the full identity gateway: issuing identities and enforcing least‑privilege on the front door.

Receipts verified: 80/80 · Violations flagged: 12 · All signatures valid

Each receipt has three independent anchors:

  • Ed25519 signature – any field change breaks the signature.
  • RFC 3161 timestamp – proves the receipt existed at that time; backdating is impossible.
  • Commitment hashes – hashes of content and provider logs; no sensitive data leaves the customer environment.

AIUC‑1 / ISO 42001 Control Mapping

Each receipt is tagged with the AIUC‑1 controls it evidences. Examples:

Receipt Field Control What It Proves
action + in_policy E015 Action Logging Every agent action logged with policy outcome.
violation_type D004 Tool Call Testing Unauthorized / out‑of‑policy actions detected and recorded.
signature_ed25519 B001 Adversarial Robustness Evidence is tamper‑evident and independently verifiable.
content_hash D003 Restrict Unsafe Tool Calls Commits to tool call content without exposing it.
source_log_hash B006 System Security Cross‑reference with provider logs detects manipulation.
tsr_token_ref E010 Acceptable Use Policy Independent timestamp proves evidence existed before dispute.
policy_id + policy_version E007 Change Management Links each action to the exact policy version in force.
agent_id B005/B007 IAM & Access Control Identifies which agent acted, traceable to authorization chain.
